# 0️⃣ Gitの構造（最重要理解）

Gitは3層構造です：

```
作業フォルダ（Working Directory）
        ↓ git add
ステージ（Index）
        ↓ git commit
履歴（Repository）
```

# 🔎 「すべて保存」の意味

Visual Studio の保存は：

```
メモリ → ファイルに書き出す
```

だけです。

Gitは：

```
ファイル → コミット（履歴に記録）
```

です。

つまり

| 操作         | 何をしているか |
| ---------- | ------- |
| Ctrl+S     | ファイル保存  |
| すべて保存      | 全ファイル保存 |
| git add    | 追跡開始    |
| git commit | 履歴に確定保存 |

# 🔥 今回なぜ消えたか

あなたは：

1. ブランチ作成
2. 修正
3. 保存（VS）
4. コミットせずにブランチ操作
5. reset / revert 実行

→ Gitは「コミットされていない変更」を保持しません。

`reset --hard` は特に危険で：

> 作業フォルダを完全にコミット状態に戻す

ので、未コミット変更は完全に消えます。

基本は **4種類のブランチ**を使います。

```
master (安定版)
develop (開発統合)
feature/*
hotfix/*
```

構造

```
master
   ↑
develop
   ↑
feature/*
```

# 2. 各ブランチの役割

## master（リリース・完全安定）

```
master
```

用途

* **常にビルド可能**
* **安定状態のみ**
* バージョンタグ付け

```

v0.2 Windowシステム
v0.3 DirectX初期化
```

ここは **壊れたコードを絶対入れない**。

```
develop
```

用途

* 機能開発の統合場所
* 日常開発のベース

流れ

```
feature → develop → master
```

---

## feature（機能開発）

```
feature/*
```

例

```
feature/GraphicDevice

```

用途

* 新機能
* リファクタ
* 実験

```
hotfix/*
```

例

```
hotfix/crash_fix
hotfix/memory_leak
```

用途

* masterの緊急バグ修正

# もし良ければ次にこれも解説できます

あなたの今の質問内容からして、ここが次の重要ポイントです。

### ② **エンジン開発でのGit安全運用（事故防止）**

具体的に

* Gitで **プロジェクトを壊さない方法**
* **絶対やってはいけない操作**
* **Git restore / reset / revert の違い**
* **履歴を壊さない運用**

これは **エンジン開発では超重要**です。
（これ知らないと一度はプロジェクト消えます）
Gitを理解する上で **「ステージ（stage）」と「stash」**はかなり重要です。
これは **Gitの内部の作業領域の仕組み**に関係しています。

まず、Gitには **3つの作業エリア**があります。

---

# Gitの3つの作業エリア

```text
作業ディレクトリ (Working Directory)
        ↓ git add
ステージ (Staging Area / Index)
        ↓ git commit
リポジトリ (Repository)
```

### ① Working Directory（作業ディレクトリ）

あなたが普段編集しているファイルです。

例

```
GraphicDevice.cpp を編集
```

この状態は **Gitの履歴にはまだ入っていません。**

---

### ② Staging Area（ステージ）

**「次のcommitに含めるファイルを選ぶ場所」**です。

コマンド

```bash
git add GraphicDevice.cpp
```

これをすると

```
Working Directory
    ↓
Staging Area
```

に移動します。

つまり

**「この変更をcommitする予定です」**

という予約状態になります。

---

### ③ Repository（履歴）

ここで初めて履歴になります。

```bash
git commit -m "GraphicDevice追加"
```

すると

```
Working Directory
      ↓
Staging
      ↓
Repository（履歴）
```

になります。

---

# 具体例

ファイル3つ修正したとします

```
GraphicDevice.cpp
Window.cpp
Scene.cpp
```

しかし commit には **1つだけ入れたい**

```
git add GraphicDevice.cpp
git commit -m "GraphicDevice追加"
```

結果

```
commit
└ GraphicDevice.cpp
```

他のファイルはまだ作業状態です。

これが **ステージの意味**です。

---

# 図で理解

```
修正
↓
Working Directory

git add
↓
Staging Area

git commit
↓
Repository
```

---

# stash（スタッシュ）

stashは

**作業途中の変更を一時保存する機能**

です。

意味は英語の

```
stash = 隠す / しまう
```

---

# 使う場面

作業中

```
GraphicDevice.cpp 編集中
```

しかし

```
別ブランチに移動したい
```

でもGitは怒ります。

```
error: changes would be overwritten
```

---

# ここで stash

```bash
git stash
```

すると

```
作業内容
↓
stashに退避
↓
Working Directory が空になる
```

---

# stashの流れ

### 退避

```bash
git stash
```

作業が隠される。

---

### ブランチ変更

```bash
git checkout master
```

---

### 作業復帰

```bash
git stash pop
```

すると

```
隠していた変更が戻る
```

---

# stashの構造

Git内部では

```
stash stack
```

という形です。

```
stash@{0}
stash@{1}
stash@{2}
```

---

確認

```bash
git stash list
```

---

# stash図

```
Working Directory
    ↓
git stash
    ↓
stash storage
```

戻す

```
git stash pop
```

---

# 開発でよくあるstashの使い方

例

```
Window実装中
```

途中で

```
バグ修正必要
```

そこで

```
git stash
git checkout hotfix
```

修正後

```
git checkout feature/window
git stash pop
```

---

# まとめ

### ステージ

```
commitする予定のファイルを置く場所
```

操作

```
git add
```

---

### stash

```
作業途中の変更を一時保存
```

操作

```
git stash
git stash pop
```

---

# 最後に（かなり重要）

Git初心者が **一番理解すると強くなる概念**はこれです。

```
Working Directory
Staging Area
Repository
```

この3つを理解すると
Gitの **90%が理解できます。**

---

もしよければですが、
あなたのログを見る限り **次に理解するとGit事故が激減する知識**があります。

**「git reset / revert / restore の完全な違い」**

これはGitで一番混乱する部分なので、
エンジン開発者向けに **図付きで解説できます。**
