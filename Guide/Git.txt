# Gitは「履歴管理」ではない

多くの人が誤解します。

Gitの本質は：

> 安心して壊せる環境を作ること

です。

---

# あなたの今のフェーズでGitが必要な理由

あなたは：

* SceneContextを分割したい
* Renderer所有を変更したい
* unique/sharedを整理したい
* クラス分割したい

これは**構造変更フェーズ**です。

このフェーズでGitがないのは危険。

---

# 実際どうすればいいか（超シンプル版）

Visual Studioで：

1. ソリューション右クリック
2. 「Git リポジトリの作成」
3. ローカルでOK

これだけ。

GitHub不要。

---

# 最低限覚えるコマンド（GUIでOK）

* Commit（保存ポイント）
* Branch（実験用分岐）
* Switch（切り替え）
* Revert（戻す）

これだけで十分。

---

# 実際の運用例（あなた向け）

## 今の状態を保存

```
Initial stable DX11 base
```

Commit。

---

## 新ブランチ作る

```
refactor/scenecontext-split
```

---

## 思い切って分解する

* SceneContext分割
* Rendererをuniqueに変更
* GraphicsDevice抽出

---

## 壊れたら？

ブランチを消せばいい。

本体は無傷。

---

# プロがなぜGitを使うのか

理由は1つ。

> 設計を恐れずに変更できるから

ファイルコピーではこれができません。

---

# 「リンク切れが怖い」問題

Gitなら：

* 変更差分が可視化される
* どの include を変えたか分かる
* どこで壊れたか追跡できる

コピー運用はブラックボックスになります。

---

# あなたの成長段階で最も重要なこと

今は：

> 設計を壊しながら学ぶフェーズ

です。

そのためのインフラがGitです。

---

# もしGitが心理的に重いなら

最低限：

* 1機能1ブランチ
* 動いたらマージ
* 1日1コミット

これだけ。

高度なことは不要。

---

# 断言します

今Gitを導入すると：

* 設計力が一段上がる
* 実験速度が上がる
* 恐怖が減る
* 改善サイクルが速くなる

導入しないと：

* 慎重になりすぎる
* 中途半端な妥協設計になる

---

# 重要な問い

あなたは今：

* Gitを「難しいもの」と思っていますか？
* それとも「面倒なもの」と思っていますか？

どちらかでアドバイスの仕方が変わります。
